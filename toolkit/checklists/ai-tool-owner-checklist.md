# AI Tool Owner Checklist

A practical checklist for the person responsible for an AI tool or AI-enabled workflow inside a small organisation.

**This checklist supports readiness and internal accountability. It is not legal advice and does not guarantee compliance.**

---

## Tool identification

- [ ] Tool name and vendor documented
- [ ] AI features identified and listed
- [ ] Use case documented: what does it help with?
- [ ] Owner assigned and informed
- [ ] Department / team using the tool listed
- [ ] Start date and renewal date recorded
- [ ] Review schedule (annual / biennial) set

## Owner responsibilities

- [ ] I understand what this tool does and its limitations
- [ ] I know who uses it and what they use it for
- [ ] I know how to raise a concern or incident
- [ ] I am familiar with the vendor's support contact
- [ ] I have read the vendor's documentation summary

## Data and access

- [ ] What customer or personal data is processed? (documented)
- [ ] Is customer data used for AI training? (confirmed with vendor)
- [ ] Is customer data shared with subprocessors? (documented)
- [ ] Who can access the tool? (defined and limited)
- [ ] What data is logged by the tool? (understood)
- [ ] Are logs reviewed regularly? (yes / no / not yet)
- [ ] Data retention settings reviewed and appropriate? (yes / no / needs work)

## Vendor documentation

- [ ] Data Processing Agreement (DPA) signed or confirmed
- [ ] Security documentation reviewed (basic summary understood)
- [ ] Vendor's AI practices documentation reviewed
- [ ] Support and incident reporting process understood
- [ ] Any red flags from initial review? (documented)

## Human review and oversight

- [ ] The tool's recommendations are reviewed before use / publication? (yes / no / not applicable)
- [ ] Decision-makers understand tool limitations? (yes / no / needs training)
- [ ] High-impact decisions require additional review? (defined)
- [ ] Tool output can be overridden or rejected? (yes / no)
- [ ] Process for requesting human override is documented? (yes / no)

## Staff guidance

- [ ] Team has received guidance on proper use? (yes / no / scheduled)
- [ ] Prohibited uses are documented? (yes / no)
- [ ] Handling errors or unexpected output is documented? (yes / no)
- [ ] Confidential / sensitive data cannot be entered? (enforced / not enforced)
- [ ] Staff know who to contact with questions? (yes / no)

## Incident reporting

- [ ] Incident reporting process defined? (yes / no)
- [ ] Who to contact if something goes wrong? (documented)
- [ ] How to escalate a concern? (documented)
- [ ] Process for documenting unusual tool behavior? (yes / no)
- [ ] Contact info for privacy / security team available? (yes / no)

## Review cadence

- [ ] Annual review scheduled? (date: _________)
- [ ] Review includes: data processing, incidents, vendor changes? (yes / no)
- [ ] Sign-off by manager or sponsor? (required / not required)
- [ ] Results documented in internal records? (yes / no)
- [ ] Next review date set? (date: _________)

## Escalation triggers

Check if any of these would require escalation to legal / privacy / security:

- [ ] New vendor practices or features?
- [ ] Change in data processing?
- [ ] Incidents or unusual errors?
- [ ] Data breach or security concern?
- [ ] Regulatory change or enforcement notice?
- [ ] Contract renewal or terms change?

## Evidence to keep

Internal documentation to retain:

- [ ] Vendor questionnaire responses (if collected)
- [ ] Risk screening form (if completed)
- [ ] Incident logs (if any)
- [ ] Review notes (annual)
- [ ] Decision records (why we use this tool)
- [ ] Any correspondence with vendor on data use or security
